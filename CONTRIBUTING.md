# 贡献指南

感谢你对 THArchive 感兴趣。这份文档说明了什么样的贡献是受欢迎的，以及哪些方向不在本项目的接受范围内。

---

## 项目归属与分工

**Yuriko** 是本项目的负责人，同时也是服务器和域名的拥有者。THArchive 站点跑在 Yuriko 的服务器上，所有涉及基础设施、部署和最终方向的决策，由 Yuriko 做出。

**Renko_1055** 负责展示策略与前端开发维护。本仓库（`THArchive`）是 Renko_1055 为了让 Yuriko 更方便地维护和操作而组织的开发目录，采用"WordPress 插件 + 主题"的结构，是考虑到项目拥有者实际操作习惯后的选择。

---

## 项目定位

THArchive 是一个东方 Project 同人接力活动的归档站，目标单一明确：收录、展示、归档活动记录。

该项目现在部署在: `https://yuriko.cn/`，由 Yuriko 个人维护，面向同人活动参与者和东方同好。


---

## 欢迎的贡献

以下类型的贡献会被认真考虑：

- **Bug 修复**：发现了功能性错误，附上复现步骤。
- **界面与交互改进**：在现有架构范围内改善用户体验。
- **文档改进**：发现文档描述有误或缺失，可以直接提 PR。
- **数据与兼容性修复**：针对特定浏览器、特定内容格式的兼容问题。
- **安全问题**：涉及 XSS、注入等安全风险，优先直接联系维护者。

---

## 不在接受范围内的内容

下面这些方向不会被合并，提前说明可以节省双方时间：

### 1. 推翻技术路线的重写提案

本项目当前采用"WordPress 主题 + 自定义插件 + Vue 局部组件"的组合方案，这是经过权衡后的**主动选择**，不是历史包袱。

不接受以下类型的提案：

- 将整个项目迁移到 Next.js / Nuxt / SvelteKit 等框架
- 将 WordPress 替换为自建后端
- 将 CPT / 后台录入流程改成 headless CMS

这类提案通常意味着完整重写，超出了本项目当前的维护能力，也不符合项目目标。如果你认为存在更好的整体方案，欢迎 fork 后自行实现。

### 2. 涉及部署基础设施的请求

本站的服务器、域名、数据库、WordPress 实例均属于项目负责人 Yuriko 的个人资产，由其自行管理。

不接受以下类型的请求：

- 将站点部署迁移到第三方服务器
- 将数据库或存储移交至外部平台
- 将域名、DNS 或 CDN 控制权转移

这类请求无论出于何种理由，都不在贡献范围内，也不会因为给出"更好的方案"而被接受。基础设施控制权归属是项目的基本前提，不是可以讨论的技术选项。

### 3. 影响核心数据结构的改动（未经讨论）

下列内容属于高风险改动，直接提 PR 大概率不会被合并：

- 修改 `relay_event` CPT 的结构
- 增删 taxonomy（`touhou_character`、`organizer`、`event_status` 等）
- 修改 meta key 名称或语义
- 修改前台投稿字段名或 REST 接口参数

如果你认为某个数据结构需要调整，请先在 Issue 中描述问题和理由，等待维护者确认后再动手。

---

## 提 Issue 的建议

- 描述你实际遇到的问题，不只是你期望的解决方案
- 复现步骤尽量具体
- 如果是功能请求，说明实际使用场景而不只是技术想法

---

## 提交 PR 的流程

1. 先开 Issue 描述你打算做什么（小 bug fix 可跳过）
2. Fork 仓库，在自己的分支上修改
3. 构建确认通过（如果改动了前端代码）：
   ```bash
   cd mydev/wp-content/plugins/tharchive-core
   npm run build
   ```
4. 提交 PR，描述改动内容和原因

---

## 最终决定权

所有功能合并、架构决策和部署安排，最终由项目负责人 **Yuriko** 决定。贡献者可以提议，但无法要求。

如果你对现有方向有根本性的不同意见，欢迎在 Issue 中理性讨论，但请理解项目有权拒绝，这不是对贡献者个人的否定。
